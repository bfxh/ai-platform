# Compound Engineering 知识系统

> 让 AI 帮你搭一个能持续生成结果的系统

## 系统目标

1. **让 AI 不要每次都从零开始**
2. **让你的知识资产以文件形式持续积累**
3. **让每一次处理、问答、产出，反过来增强系统本身**
4. **让系统能定期发现问题，而不是越用越乱**

## 核心原则

**不要只让 AI 生成一次性的结果。要让 AI 帮你生成一套能持续使用的系统。**

## 三个基本原则

1. **文件优先**：所有东西尽量落成普通文件，避免锁在单一应用里
2. **增量更新**：只处理新增内容，不要每次重建全库
3. **可追溯**：任何结论都能追溯到原始来源

## 目录结构

```
knowledge/
├── raw/                      # 原始摄取层
│   └── {date}-{source-slug}.md
├── knowledge/                 # 知识消化层
│   ├── summaries/            # 结构化摘要
│   │   └── {date}-{slug}.md
│   ├── concepts/            # 概念卡片 (Zettelkasten)
│   │   └── {concept-slug}.md
│   ├── topics/               # 主题导航页 (MOC)
│   │   └── {topic-slug}.md
│   └── index/               # 总索引
│       └── index.md
├── outputs/                  # 输出层
│   ├── article/             # 长文输出
│   ├── infographic/          # 信息图输出
│   ├── pdf/                 # PDF 输出
│   ├── ppt/                 # PPT 输出
│   ├── qa/                  # 问答输出
│   ├── memo/                # 研究备忘录
│   └── social/             # 社交媒体输出
├── inspection/             # 巡检报告
│   └── {date}-report.md
└── skills/                 # 知识系统 skills
    ├── ingest.md           # 摄取 skill
    ├── digest.md           # 消化 skill
    ├── output.md           # 输出 skill
    └── inspect.md          # 巡检 skill
```

## 四步工作流

### 📥 第一步：摄取 (Ingest)

**目标**：把外部输入统一转换成一个稳定、干净、可处理的原始层

AI 在摄取阶段只做四件事：
1. 抽取正文
2. 清理噪音
3. 保留元数据
4. 输出标准化文件

**原则**：
- 保留原文，不要过早压缩信息
- 保留来源、时间、作者、链接
- 统一文件格式
- 不做复杂判断，不做深度总结

### 🔄 第二步：消化 (Digest)

**目标**：把原始材料编译成可以复用的知识结构

AI 在消化阶段做四件事：
1. 为每份 raw 生成结构化摘要
2. 抽取概念，并映射到已有概念库
3. 更新主题页或创建新主题页
4. 更新总索引

**原则**：
- 增量方式处理新增内容
- 每条结论尽量带来源
- 先抽象出概念，再组织主题
- 人负责判断，AI 负责整理和联结

### 📤 第三步：输出 (Output)

**目标**：让每一次高质量问答、每一次内容生成都落成文件，重新变成系统资产

**原则**：
- 先检索，再综合，再生成
- 输出结果尽量引用来源
- 高价值输出必须落文件
- 不把聊天记录当资产，文件才是资产

### 🔍 第四步：巡检 (Inspect)

**目标**：定期审计系统，发现并修复结构问题

巡检应检查：
- 是否存在定义冲突
- 是否存在重复概念
- 是否存在没有被引用的孤岛文件
- 是否存在缺少来源支持的结论
- 是否存在长期未更新但仍被高频引用的内容

**原则**：
- 巡检先出报告，不要默认自动改
- 巡检结果也要落文件

## Schema 定义

### 摄取文件 Schema

```yaml
source: 来源
url: 原始链接
date: 摄取日期
author: 作者
type: 类型 (article/tweet/video/podcast/doc)
raw_content: |
  原始正文
```

### 摘要 Schema

```yaml
source: 来源
url: 原始链接
date: 摘要日期
summary: |
  结构化摘要（3-5个要点）
key_findings:
  - 要点1
  - 要点2
related_concepts:
  - 概念1
  - 概念2
```

### 概念卡 Schema

```yaml
concept: 概念名称
aliases: [别名1, 别名2]
definition: 一句话定义
source: 来源
date_created: 创建日期
related_concepts:
  - 概念1
  - 概念2
notes: |
  详细笔记
```

### 主题页 Schema

```yaml
topic: 主题名称
description: 主题描述
related_concepts:
  - 概念1
  - 概念2
key_summaries:
  - summary1.md
  - summary2.md
last_updated: 最后更新
```

## 方法论对比

| 方法 | 适用场景 | 核心思路 |
|------|----------|----------|
| PARA | 项目推进、个人管理 | 按可执行性组织 |
| MOC | 主题研究、知识浏览 | 用导航页组织 |
| Zettelkasten | 深度研究、概念沉淀 | 原子化笔记+链接 |

## 常见误区

1. **一上来就让 AI 直接写最终内容** → 会得到稿子，但得不到系统
2. **把聊天记录当知识资产** → 聊天记录只是过程，文件才是资产
3. **分类过细、链接过多** → 维护成本会迅速上升
4. **过早上 RAG** → 规模不大时，良好的目录结构通常已经够用
5. **让 AI 静默修改知识库** → 没有审计和可追溯性，系统会越来越不可信

## 最小闭环

1. 摄取一篇文章和一条推文，存到 raw/
2. 对这两个 raw 文件执行一次消化
3. 回答一个具体问题，并把回答存到 outputs/qa/
4. 对 knowledge/ 做一次巡检，生成报告

> 不要一开始就搞大而全，先跑通最小闭环，再慢慢增强。
