# 怎么跟AI讲开发需求 - 从这次对话总结的最佳实践

## 一、说清楚"做什么"，别说"搞一下"

**差的说法：**
- "帮我搞一下MCP" → AI不知道搞什么
- "优化一下" → 优化哪里？性能？代码？架构？
- "增加功能" → 什么功能？给谁用？

**好的说法：**
- "在/python/MCP下创建一个网络工具MCP服务，支持ping/DNS查询/端口扫描，用纯Python不要第三方依赖"
- "mcp_coordinator.py的跨MCP请求转发超时，协调器转发请求后目标服务的RESPONSE没有回传唤醒等待线程，修复这个消息路由问题"
- "把所有MCP服务合并到一个进程里运行，不要开多个终端窗口，提供Web UI管理"

**模板：**
```
在 [位置] 创建/修改 [什么东西]
功能：[具体功能列表]
约束：[技术限制/不要做什么]
输出：[期望的文件/格式/效果]
```

---

## 二、一次说一件事，说完验证再说下一件

这次对话的教训：一次提太多需求（"增加功能+优化+修bug+搞UI"），AI容易顾此失彼。

**好的节奏：**
1. "创建网络MCP服务" → 等AI做完 → 验证能跑
2. "现在加提示词MCP" → 等AI做完 → 验证能跑
3. "把它们整合到一个进程" → 等AI做完 → 验证能跑

**坏的节奏：**
- "增加网络MCP、提示词MCP、Brain系统、UI面板、安装器、读取器，全部要生产级优化" → AI一次写太多，bug多，验证困难

---

## 三、给AI足够的上下文

AI不记得上一轮对话。每次新对话开始：

**必须告诉AI的：**
- 项目在哪：`/python/MCP`
- 核心文件：`mcp_core.py` 是主入口
- 架构：单进程多线程，协调器+内嵌服务+WebUI
- 你的规矩：不要开多终端、做完要验证、简洁不废话

**可以这样开头：**
```
项目在 /python，核心是 /python/MCP/mcp_core.py（单进程MCP系统）。
先读 mcp_core.py 了解架构，然后帮我 [具体需求]。
做完用 py_compile 验证语法，不要开新终端窗口。
```

---

## 四、出错时怎么说

**差的说法：**
- "不行，再试试" → AI不知道哪里不行
- "有bug" → 什么bug？什么现象？

**好的说法：**
- "运行 python mcp_core.py 后，19801端口没有监听，日志显示 [具体错误]"
- "调用 /api/call?tool=brain_think 返回404，但 /api/health 正常，说明路由表缺少 /api/call 端点"

**模板：**
```
操作：[我做了什么]
期望：[应该发生什么]
实际：[实际发生了什么]
日志：[关键错误信息]
```

---

## 五、让AI读文件而不是你复制粘贴

**差的做法：** 把整个文件内容粘贴到对话里（浪费token，容易截断）

**好的做法：**
- "先读 /python/MCP/mcp_core.py 的第850-900行，那里是服务注册逻辑"
- "grep一下 mcp_core.py 里所有包含 'register' 的行"
- "读 mcp-config.json 了解当前有哪些服务"

---

## 六、验证清单

每次AI说"完成了"，要求它：
1. `py_compile` 检查语法
2. 实际运行一下看有没有报错
3. 调用一个功能确认能用
4. 如果改了多个文件，全部检查

可以直接说：
```
做完后：1.语法检查 2.启动测试 3.功能验证
```
