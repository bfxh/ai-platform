# 本地化 AI 能力集成平台文档

## 1. 项目概述

一个本地运行的调度平台，统一连接和管理多种形态的能力单元，包括：
- Skill（单一功能脚本）
- MCP 服务（Model Context Protocol 本地服务）
- CLI 工具
- Workflow（有向无环图编排）
- Agent（带记忆/规划的智能体）
- Plugin（复合集成模块）
- 本地模型文件（GGUF / ONNX / PyTorch 等）

### 核心设计
1. **形态无关，皆可集成**：提供描述文件 + 标准入口，平台即可加载
2. **本地执行**：所有能力在本机执行，数据不出本机
3. **MIT 协议**
4. **沙盒执行**：所有能力默认在沙盒环境执行，由 TRAE 监控
5. **超时处理**：单次任务超过 10 分钟，自动保存现场到"垃圾堆"
6. **自动化中枢**：TRAE 负责安装、配置、调度、超时保存、每日存储提交

## 2. 能力单元形态与接入约定

| 形态 | 说明 | 接入方式 |
|------|------|----------|
| Skill | 单一功能脚本或微服务 | skill.yaml + 入口脚本 |
| MCP 服务 | 符合 Model Context Protocol 的本地服务 | mcp.json，启动命令 + 环境变量 |
| CLI 工具 | 命令行可执行程序 | 可执行路径 + 参数模板 |
| Workflow | 有向无环图编排多个能力 | 工作流定义文件（YAML/JSON） |
| Agent | 带记忆/规划的自主智能体 | agent.yaml，含循环与工具调用权限 |
| Plugin | 复合集成模块，可内含上述形态 | 独立目录，含 plugin.toml |
| 本地模型 | GGUF / ONNX / PyTorch 等 | 路径 + 推理脚本，由 Skill/Agent 调用 |

**游戏测试约定**：最终封装为 Plugin。Plugin 内部通过 CLI 启动游戏，MCP 验证状态，Skill 执行操作。

## 3. 分层架构

四层架构：用户层、项目层、适配层、存储层。不设通用层。

### 3.1 用户层

划分三个区域：
- **个人数据区**：用户项目、配置、笔记、密钥，完全由用户自主管理
- **运行时保障数据区**：临时密钥、缓存、状态恢复数据，平台自动维护
- **全局项目层**：环境基础能力，本质不变，个人调整仅在个人数据区覆盖

**全局项目层包含**：
| 目录 | 内容 | 约束 |
|------|------|------|
| skill/ | Skill 描述文件与入口脚本 | 本质不变 |
| agent/ | Agent 描述文件与配置 | 本质不变 |
| plugin/ | Plugin 描述文件与插件目录 | 本质不变 |
| rag/ | RAG 配置与向量库路径 | 本质不变 |
| prompt/ | 系统级 Prompt 模板 | 本质不变 |
| context/ | 上下文管理配置 | 本质不变 |
| multi-agent/ | 多智能体编排配置 | 本质不变 |
| models/ | 模型路径映射与推理脚本 | 本质不变 |

### 3.2 项目层

每个项目分为两部分：
1. **框架内部分**（必定存在）：项目声明、配置、索引文件，位于平台管理下
2. **外部工作区**（路径自定义）：实际软件、插件、脚本等实体，由用户自行指定

**项目必须提供**（框架内）：
- README.md：项目教程
- INDEX.md：能力单元列表与链接
- BUGS.md：Bug 记录
- plugin.toml 或 manifest.yaml：入口描述，必须声明外部工作区路径
- 环境配置：环境变量清单（名、用途、示例值，不含真实密钥）
- 软件配置：依赖的本地软件名称、外部工作区中的相对路径、版本

**项目约束规则**：
1. 目录独立性：每个项目在 projects/ 下拥有唯一子目录，名称全小写，可含连字符，禁止空格
2. 编码与格式：所有文本文件使用 UTF-8 编码，无 BOM。描述文件为 TOML / YAML / JSON 之一，键名统一小写下划线
3. 路径声明强制：plugin.toml 必须包含 [workspace] 表，external_path 为绝对路径或相对于项目目录的相对路径
4. 环境配置规范：环境变量清单以 [environment] 数组列出，每项包含 name、description、example，不得包含真实密钥
5. 软件配置规范：[software] 表下列出依赖的可执行程序，给出相对外部工作区的路径、最低版本、可选参数模板
6. 能力绑定：项目通过 plugin.toml 的 [[abilities]] 数组列出提供的所有能力单元
7. 索引强制：INDEX.md 必须列出项目中每个能力单元的名称、形态、简要说明和指向其描述文件的链接
8. Bug 记录格式：BUGS.md 每条记录格式统一，含时间戳、状态、摘要、复现步骤
9. 教程强制：README.md 必须包含"安装与接入""配置说明""使用示例"三个章节
10. 版本管理独立性：框架内文件纳入平台 Git 管理。外部工作区文件不纳入，由用户自行版本控制
11. 独立可迁移：项目框架内目录可完整复制到其他平台实例，仅需修改 external_path 即可重新加载

### 3.3 适配层

**职责**：
- 操作系统适配：统一路径格式、环境变量命名、系统调用方式
- 软件适配：解析项目 external_path，扫描外部工作区，生成路径映射和参数模板，供调度使用
- 自身提供 README.md（教程）和 INDEX.md（索引），变更时同步更新 TRAE 配置和存储层映射

### 3.4 存储层

**统一存放**：
- 各能力单元的描述文件（skills/、mcps/、clis/、workflows/、agents/、plugins/）
- 每日摘要 JSON（daily/YYYY-MM-DD.json）
- 垃圾堆快照（trash/）
- 自身提供 README.md 和 INDEX.md

## 4. Git 管理

**纳入版本库**：
- 插件核心代码
- 所有能力单元的配置、入口脚本
- 项目框架内全部文件
- 软件目录映射
- 每日存储摘要
- 全局项目层

**不纳入版本库**（.gitignore）：
- 第三方软件本体、模型文件、安装包
- 用户敏感数据（密钥、token）
- 运行时缓存、临时文件
- 大型二进制数据（改用 Git LFS 或路径引用）
- 垃圾堆目录（可选择性 LFS）
- 项目声明的外部工作区路径及其内部文件

## 5. 垃圾堆机制

- 超时 10 分钟，TRAE 打包沙盒状态为 storage/trash/YYYY-MM-DD_HHMMSS_taskid.snapshot
- 快照含运行日志、内存变量摘要、文件系统差异清单
- 快照可 Git LFS 提交，原始数据留临时目录
- 垃圾堆不自动清理，用户决定处理

## 6. TRAE 自动化中枢职责

- 安装向导：初始化 Git，写入 .trae/config.toml
- 连接配置：扫描项目外部工作区路径，生成映射，更新存储层
- 每日存储面板：日历视图，一键提交
- 任务监控：沙盒执行，超时挂起存垃圾堆，通知用户
- 架构同步：目录或规范变更时自动更新调度逻辑
- 自身配置和日志不可手动任意修改

## 7. 游戏测试流程（Plugin 集成）

以 WorldCraft 为例：
1. 框架内声明：projects/worldcraft_test/plugin.toml，声明 external_path = "../wc_tools"
2. 定义 CLI worldcraft.exe、Skill explore_skill.py、MCP game_state_mcp、Workflow 编排
3. 加载与执行：
   - TRAE 读取项目，适配层扫描外部工作区 wc_tools/ 找到实体
   - 启动沙盒，CLI 进入游戏，Skill 操作，MCP 验证，异常写入 BUGS.md
   - 超时存垃圾堆

## 9. 规则分类

### 9.1 核心运行规则
1. 所有能力本机执行，数据不出本机
2. 默认沙盒执行
3. 自动化任务非显式允许不得删除文件
4. 超时 10 分钟自动存垃圾堆
5. MIT 协议

### 9.2 能力单元接入规则
1. 形态无关，符合描述即加载
2. 各形态必须提供规定接入文件
3. 游戏测试须封装为 Plugin

### 9.3 分层规则
1. 四层：用户、项目、适配、存储，无通用层
2. 用户层三区隔离
3. 全局项目层存放不变基础能力
4. 项目层分框架内与外部工作区，外部路径自定义
5. 适配层负责路径解析、OS 差异、软件对接
6. 存储层统一存放描述与快照
7. 所有层均含教程和索引

### 9.4 项目约束规则
1. 目录命名小写连字符，无空格
2. 文本 UTF-8 无 BOM，描述文件键名小写下划线
3. plugin.toml 必须声明 [workspace] external_path
4. 环境配置不含真实密钥，仅名/描述/示例
5. 软件配置用相对外部工作区的路径，适配层校验存在性
6. 能力列表在 [[abilities]] 中完整声明
7. INDEX.md 即时同步所有能力
8. BUGS.md 格式统一，含时间戳、状态、摘要、复现步骤
9. README.md 含三个必含章节
10. 框架内文件属平台 Git，外部工作区独立版本控制
11. 项目目录整体可迁移

### 9.5 Git 版本规则
1. 代码、配置、教程、索引、Bug 记录均入库
2. 软件本体、密钥、缓存、大二进制不入库
3. 外部工作区不入平台 Git
4. 可上传 GitHub

### 9.6 垃圾堆与超时处理规则
1. 超 10 分钟即存快照
2. 快照含日志、变量摘要、文件差异
3. 可选 Git LFS，原始数据留临时目录
4. 垃圾堆不自动清理

### 9.7 TRAE 自动化中枢规则
1. 安装初始化 Git 和 config.toml
2. 扫描项目外部路径生成映射
3. 每日面板一键提交
4. 沙盒监控，超时挂起，通知用户
5. 架构变更同步自身
6. 自身配置不可手动随意改

### 9.8 适配层规则
1. 仅处理 OS 差异和外部软件对接
2. 路径格式、环境变量、系统调用统一
3. 扫描外部工作区，生成映射
4. 变更同步更新 TRAE 和存储层

## 10. 总结

本地化 AI 能力集成平台提供了一个统一的框架，用于管理和调度多种形态的 AI 能力单元。通过严格的分层架构和规则体系，确保了平台的可扩展性、安全性和可维护性。平台的设计理念强调本地执行、形态无关、沙盒隔离等核心原则，为 AI 能力的集成和管理提供了一个灵活而强大的解决方案。
